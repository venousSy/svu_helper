WHAT CHANGED AND WHY
================================================================================
The developer wrote one Python file called bot.py that does everything:
sets up the database, defines all bot handlers, and runs the bot.

The database is SQLite (Python's built-in database) with a single table called
"requests" that has four columns: an auto-incrementing ID, the student's
Telegram user_id, their message text, and a status field that defaults to
"Pending".

There are exactly three handlers:

1. /start command → sends a greeting message to whoever opens the bot.

2. handle_user_request (catches ALL messages from non-admins):
   - Saves the student's message to SQLite as a new row with status "Pending"
   - Sends the student a confirmation with their request ID number
   - Forwards a notification to the admin's Telegram chat (hardcoded ADMIN_ID)
     that includes the request ID and the student's message

3. admin_reply_handler (only fires when the admin REPLIES to a notification):
   - Parses the request ID from the notification text by splitting on "ID: #"
   - Looks up the student's user_id in the database using that ID
   - Sends the admin's reply text directly to that student's chat
   - Updates the database row status to "Resolved"

The admin workflow is entirely reply-based: the admin reads the notification
message in their Telegram chat, writes a reply, and the bot forwards it to the
right student. There are no buttons, no menus, no inline keyboards yet.

Notable problems that will be fixed in future commits:
- The Telegram API token is hardcoded in plain text in the source file
- The admin's Telegram ID is a magic number (7450958767) hardcoded in the file
- The SQLite database file (bot_requests.db) is committed directly to git
- All database connections are opened and closed inline with no abstraction layer
- There is no error handling if the database operation fails
- There is no way for students to check the status of their previous requests
- The admin can only reply to one request type — there are no offer/price flows yet

import asyncio
import sqlite3  # Built-in, no install needed
from aiogram import Bot, Dispatcher, types
from aiogram.filters import Command

ADMIN_ID = 7450958767

# --- DATABASE LOGIC ---
def init_db():
    conn = sqlite3.connect("bot_requests.db")  # Creates the file if it doesn't exist
    cursor = conn.cursor()
    cursor.execute('''
        CREATE TABLE IF NOT EXISTS requests (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            user_id INTEGER,
            message_text TEXT,
            status TEXT DEFAULT 'Pending'
        )
    ''')
    conn.commit()
    conn.close()

init_db()

API_TOKEN = '8395569134:AAHun2cnEbo8YJ39Y08cEKH2AMvz1_HBrZI'  # hardcoded token

bot = Bot(token=API_TOKEN)
dp = Dispatcher()

@dp.message(Command("start"))
async def welcome(message: types.Message):
    await message.answer("Hello! Send me a request.")

@dp.message(lambda message: message.from_user.id == ADMIN_ID and message.reply_to_message)
async def admin_reply_handler(message: types.Message):
    original_text = message.reply_to_message.text
    try:
        req_id = original_text.split("ID: #")[1].split("\n")[0]
    except Exception:
        await message.answer("❌ Error: Could not find the Request ID in the replied message.")
        return

    conn = sqlite3.connect("bot_requests.db")
    cursor = conn.cursor()
    cursor.execute("SELECT user_id FROM requests WHERE id = ?", (req_id,))
    result = cursor.fetchone()

    if result:
        user_to_reply = result[0]
        await bot.send_message(chat_id=user_to_reply, text=f"📩 **Reply from Admin:**\n{message.text}")
        cursor.execute("UPDATE requests SET status = 'Resolved' WHERE id = ?", (req_id,))
        conn.commit()
        await message.answer(f"✅ Reply sent to User {user_to_reply}!")
    else:
        await message.answer("❌ Could not find that request in the database.")
    conn.close()

@dp.message()
async def handle_user_request(message: types.Message):
    uid = message.from_user.id
    text = message.text
    if message.from_user.id == ADMIN_ID:
        return

    conn = sqlite3.connect("bot_requests.db")
    cursor = conn.cursor()
    cursor.execute(
        "INSERT INTO requests (user_id, message_text) VALUES (?, ?)",
        (uid, text)
    )
    request_id = cursor.lastrowid
    conn.commit()
    conn.close()

    await message.answer(f"✅ Your request (ID: {request_id}) has been saved and is pending review.")

    notification_text = (
        f"🔔 **NEW REQUEST**\n"
        f"ID: #{request_id}\n"
        f"User: {message.from_user.full_name}\n"
        f"Message: {text}\n\n"
        f"Reply to this message to send your answer!"
    )
    await bot.send_message(chat_id=ADMIN_ID, text=notification_text)

async def main():
    print("Bot is running...")
    await dp.start_polling(bot)

if __name__ == "__main__":
    asyncio.run(main())

import sqlite3

def read_all_requests():
    conn = sqlite3.connect("bot_requests.db")
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM requests")
    rows = cursor.fetchall()
    conn.close()
    return rows

if __name__ == "__main__":
    requests = read_all_requests()
    if not requests:
        print("No requests found in the database.")
    else:
        print(f"{'ID':<5} {'User ID':<15} {'Message':<40} {'Status':<15}")
        print("-" * 75)
        for row in requests:
            print(f"{row[0]:<5} {row[1]:<15} {str(row[2])[:37]:<40} {row[3]:<15}")

================================================================================
KEY DESIGN DECISIONS (and why they will change later)
================================================================================

Decision: Admin identification by reply-to-message
  → The admin has no special command or panel. They just reply to bot
    notifications. This works for one admin with low volume but breaks as soon as
    there are multiple admins or the notification thread gets long.
  → CHANGED IN: commit #11 (button-based offer system)

Decision: SQLite as the database
  → Simplest possible persistence — no server needed, just a file. But SQLite
    does not support concurrent writes well and cannot be used across multiple
    processes or containers.
  → CHANGED IN: commit #59 (MongoDB migration)

Decision: Token hardcoded in source
  → Fast to get started, but a security disaster — anyone who sees the code gets
    full control of the bot.
  → CHANGED IN: commit #43 (centralized constants + .env file)

Decision: Everything in one file
  → Easy to reason about when the codebase is 109 lines. Impossible once it grows.
  → CHANGED IN: commits #5-6 (modularization into client.py and admin.py)
